# 各实验数据与处理思路

## `build_global_face_oval_meanbg.py`：全局人脸椭圆区域 + VGGFace2 均值背景

### 1. 处理目标

该脚本构建一套用于 Baseline3 v2（Version C）的全局人脸图像数据。核心思路是：

> 先检测、裁剪并对齐人脸，再利用 MediaPipe FaceMesh 提取脸部椭圆区域；保留椭圆内的人脸信息，将头发、衣服、拍摄环境等椭圆外内容替换为 VGGFace2 训练集均值颜色。

这样处理的主要目的，是在保留较完整面部表型信息的同时，减少背景、服装、发型和拍摄环境等非目标因素对模型的干扰，并使生成图像与采用 VGGFace2 预训练权重的模型输入分布更加协调。

脚本会写入新的数据目录，不修改已有的 `global_224` 目录和 `processed_features.csv`。

### 2. 输入数据与样本组织

- 原始图像目录：`E:\projects\face\data\raw\images`
- 标签文件：`E:\projects\face\data\raw\label.csv`
- 支持格式：`.jpg`、`.jpeg`、`.png`、`.bmp`
- 标签表至少需要包含：
  - `ID`：样本标识，同时用于查找同名图片；
  - `NYHA`：分类标签，主要用于质控预览的分层抽样。

数据收集有两种模式：

1. **标签表可正常读取时**  
   按 `label.csv` 的行顺序遍历样本，依次尝试寻找 `ID + 扩展名` 对应的图片。即使某个 ID 没有图片，也会保留该样本并在处理日志中记录失败。

2. **标签表不可用时**  
   直接扫描原始图片目录，将文件名去掉扩展名后作为 ID；此时 NYHA 标签记为空值。

这种设计使标签表既承担标签来源的作用，也承担预期样本清单的作用，便于发现标签存在但原图缺失的样本。

### 3. 单张图像的预处理流程

整体流程如下：

```text
原始 BGR 图像
  → MediaPipe 人脸检测
  → 扩展检测框并裁剪人脸
  → 基于双眼位置进行旋转对齐
  → 缩放为 224×224
  → FaceMesh 提取脸部椭圆轮廓
  → 生成并羽化椭圆掩膜
  → 椭圆外替换为 VGGFace2 均值背景
  → JPEG 保存
```

#### 3.1 人脸检测与扩展裁剪

脚本调用 `detect_and_crop_face()`，底层使用 MediaPipe Face Detection：

- `model_selection=1`；
- 最低检测置信度为 `0.5`；
- 只使用返回的第一张人脸。

在原始检测框基础上进行扩展：

- 上方扩展检测框高度的 10%，用于保留额头；
- 下方扩展检测框高度的 20%，用于保留下颌；
- 左右分别扩展检测框宽度的 20%，用于保留面颊；
- 所有坐标限制在原始图像边界内。

因此，该步骤不是严格贴合检测框裁剪，而是先保留一定的人脸周边区域，为后续对齐和椭圆分割留下空间。

如果未检测到人脸，该样本直接记为失败，不生成输出图像。

#### 3.2 仿射对齐与尺寸统一

裁剪后调用 `affine_alignment()`：

1. 使用 FaceMesh 定位左右眼内外眼角；
2. 分别计算左右眼中心；
3. 根据双眼中心连线计算倾斜角度；
4. 以双眼中点为旋转中心，将眼线旋转至水平；
5. 旋转产生的边缘区域采用 `BORDER_REFLECT_101` 镜像填充，避免黑边；
6. 使用 Lanczos4 插值缩放到 `224×224`。

这里的“仿射对齐”主要是**平面内旋转校正 + 尺寸统一**，没有进一步固定双眼间距、眼睛纵向位置或进行三点仿射归一化。因此，它能降低头部倾斜带来的差异，但不能完全消除人脸在画面中的平移和尺度变化。

如果该阶段的 FaceMesh 未检测到关键点，底层函数会退化为直接将裁剪图缩放至 `224×224`，而不是判定处理失败。

#### 3.3 FaceMesh 脸部椭圆掩膜

在对齐后的 `224×224` 图像上再次运行 MediaPipe FaceMesh：

- 静态图像模式；
- 最多检测 1 张人脸；
- 启用 `refine_landmarks=True`；
- 最低检测置信度为 `0.5`。

脚本使用固定的 36 个 FaceMesh 轮廓点，按顺序连接成人脸椭圆多边形。关键点的归一化坐标转换为像素坐标后，会被限制在图像边界内。随后：

1. 用白色填充脸部多边形，生成二值掩膜；
2. 计算二值掩膜在整张 `224×224` 图像中的面积占比 `mask_area_ratio`；
3. 使用 `9×9` 高斯核、`sigmaX=3` 对掩膜进行羽化。

羽化的作用是使脸部边缘与背景之间平滑过渡，避免硬切割产生明显锯齿或轮廓边界。

如果在对齐图像上无法检测到 FaceMesh，该样本记为失败。

#### 3.4 VGGFace2 均值背景替换

VGGFace2 的 RGB 通道均值为：

```text
RGB = [131.0912, 103.8827, 91.4953]
```

由于 OpenCV 使用 BGR 通道顺序，脚本实际填充的背景颜色为：

```text
BGR = [91.4953, 103.8827, 131.0912]
```

羽化掩膜归一化为 `alpha∈[0,1]` 后，逐像素混合：

```text
输出图像 = 对齐图像 × alpha + VGGFace2均值背景 × (1 - alpha)
```

其中：

- 椭圆内部 `alpha` 接近 1，主要保留原始人脸像素；
- 椭圆外部 `alpha` 接近 0，替换为固定均值背景；
- 椭圆边缘为连续过渡。

使用均值颜色而不是黑色或白色背景，有两个考虑：

1. 减少高对比度人工边界；
2. 如果后续模型执行 VGGFace2 均值减法，背景区域将接近零响应，降低背景对特征提取的影响。

需要注意，当前脚本本身只生成 `uint8` BGR JPEG，并没有执行模型输入阶段的均值减法或归一化。

### 4. 输出结果

- 处理后图像目录：  
  `E:\projects\face\data\processed\global_face\global_face_oval_meanbg_224`
- 处理日志：  
  `E:\projects\face\log\global_face_oval_meanbg_224_preprocess_log.csv`
- 质控预览图：  
  `E:\projects\face\data\processed\global_face_oval_meanbg_224_preview.jpg`

输出图片统一为：

- 尺寸：`224×224`；
- 通道：OpenCV BGR；
- 格式：JPEG；
- JPEG 质量：95；
- 文件名：`ID.jpg`。

### 5. 日志与质量控制

每个样本都会生成一条日志，主要字段包括：

- `ID`
- `input_path`
- `output_path`
- `status`
- `error_message`
- `face_detected`
- `facemesh_detected`
- `mask_area_ratio`
- 原始图像宽度和高度

该日志可以用于统计处理成功率，并定位以下问题：

- 原图缺失或无法读取；
- 人脸检测失败；
- FaceMesh 椭圆检测失败；
- 输出文件写入失败；
- 椭圆面积异常。

脚本还会生成 `5` 列的预览网格：

- 标签可用时，对 NYHA=0 和 NYHA=1 分别最多随机抽取 20 张；
- 无法按标签分层时，从成功样本中最多随机抽取 40 张；
- 固定随机种子为 `42`，保证重复运行时抽样结果一致；
- 每张图下方标注样本 ID 和 NYHA 标签。

预览图适合人工检查对齐方向、椭圆边界、脸部截断、背景替换及类别间是否存在明显的数据质量差异。

### 6. 该方案保留与抑制的信息

**主要保留的信息：**

- 脸部整体外观；
- 眼、鼻、口、面颊、额头和下颌区域；
- 面部肤色、纹理和形态；
- 一定程度的表情与局部非对称信息。

**主要抑制的信息：**

- 图像原始背景；
- 衣服和肩部；
- 大部分头发、耳部及脸部轮廓外信息；
- 拍摄场景和构图差异；
- 可能与标签偶然相关的背景偏差。

因此，该数据版本适合验证：模型性能是否主要来自脸部区域，而不是背景、衣着或其他非面部线索。它可以与保留完整裁剪图的 `global_224` 数据形成对照实验。

### 7. 实现上的注意事项与局限

1. **项目路径被硬编码**  
   脚本中的 `PROJECT_ROOT` 为 `E:\projects\face`，不是当前仓库 `E:\projects\face2`。运行前应确认这是有意读取另一项目的数据和工具代码，否则可能在错误目录中读写数据。

2. **未执行颜色校正**  
   虽然 `image_utils.py` 中存在 `color_correction()`，本脚本没有调用它。因此不同设备、光照和白平衡造成的颜色差异仍会保留在人脸椭圆内。

3. **对齐不是完整几何归一化**  
   当前仅校正双眼连线倾斜并整体缩放，没有将关键点映射到统一模板位置。样本间仍可能存在脸部大小和中心位置差异。

4. **对齐关键点失败不会写入日志**  
   对齐函数在 FaceMesh 失败时直接缩放，主脚本无法区分“成功完成旋转对齐”和“退化为直接缩放”。现有日志中的 `facemesh_detected` 只表示后续椭圆掩膜阶段是否检测成功。

5. **只处理第一张检测人脸**  
   若原图包含多人，脚本默认使用 MediaPipe 返回的第一张人脸，未根据面积、中心位置或身份进一步筛选。

6. **椭圆区域不等于严格皮肤区域**  
   FaceMesh 椭圆会包含眉毛、眼睛、嘴唇等全部脸内结构，也可能保留少量额前头发；它的目的只是限制为整体面部区域，不是进行皮肤语义分割。

7. **缺少面积阈值过滤**  
   `mask_area_ratio` 仅被记录，没有设置合理范围或自动剔除规则。关键点异常但仍形成多边形时，可能生成面积过大、过小或形状异常的掩膜，需要结合日志分布和预览图检查。

8. **重复运行可能保留旧文件**  
   输出目录只会被创建，不会在运行前清空。若样本在旧一次运行中成功、在新一次运行中失败，旧输出图仍可能留在目录中。后续训练集构建应以当次日志中 `status=success` 的样本为准，或运行前显式管理输出目录。

9. **JPEG 会引入轻微压缩误差**  
   质量 95 通常影响较小，但如果后续研究依赖非常细微的颜色或纹理差异，可考虑无损 PNG，并评估存储成本。

### 8. 总结

该脚本构建的是一种“面部区域保留、非面部区域去偏”的标准化输入。其关键变量控制逻辑为：

```text
统一尺寸和方向
    + 保留 FaceMesh 椭圆内的面部表型
    + 将椭圆外区域固定为预训练数据均值颜色
    = 降低背景混杂的全局人脸输入
```

实验解释上，若该版本与完整全局人脸图像相比性能相近，说明模型判别信息主要来自面部椭圆区域；若性能明显下降，则提示头发、耳部、颈肩、背景或更完整的头部形态可能包含有效信息，也可能意味着椭圆分割和对齐过程损失了部分有效特征。

## `preprocess_single_roi_dataset.py`：基于 FaceMesh 的单样本多局部 ROI 数据集

### 1. 处理目标

该脚本用于从正面或近正面人脸图像中提取局部 ROI 数据集。它只执行 ROI 预处理，不训练模型，也不创建或修改训练/验证五折划分。

需要特别注意的是，该脚本当前并不是“全局人脸 ROI”提取程序，而是一个“局部区域 ROI”提取程序。每张输入图像会被拆分为四类局部区域：

- 眼周 ROI：双眼、上下眼睑、眼袋及内外眦周围皮肤；
- 唇周 ROI：上下唇、嘴角、口周皮肤、部分鼻下和下唇下方区域；
- 双侧脸颊 ROI：左脸颊与右脸颊分别裁剪后横向拼接；
- 额头 ROI：眉毛上方的下额头至中额头区域，尽量避开发际线、头发和背景。

因此，该方案适合构建局部解剖区域输入，用于分析模型是否能够从眼周、口唇、脸颊、额头等局部面部特征中学习 NYHA 相关信息；它不等价于保留整张脸的 global face 输入。

### 2. 输入数据与路径组织

脚本默认路径为：

```text
input images: data/raw/images
labels:       data/raw/label.csv
output:       data/processed/roi_dataset
```

脚本会从标签表中自动识别样本 ID、性别、二分类标签和图像路径字段。支持的候选列包括：

- patient id：`patient_id`、`PatientID`、`patientID`、`ID`、`id`
- sex：`sex`、`SEX`、`gender`、`Gender`
- binary label：`nyha_binary`、`NYHA_binary`、`y`、`label`
- fallback NYHA：`NYHA`、`nyha_grade`
- image path：`image_path`、`ImagePath`、`path`、`filename`、`file_name`、`image_name`

如果标签表中存在图像路径列，脚本会优先使用该路径；如果没有，则在输入图像目录下按 `patient_id + 扩展名` 查找图像。支持的图像扩展名包括：

```text
.jpg, .jpeg, .png, .bmp, .tif, .tiff
```

脚本支持传入预计算 landmark 目录。如果 `--landmark_dir` 中存在与样本 ID 同名的 `.npy`、`.json` 或 `.csv` 文件，则优先读取该 landmark；否则现场调用 MediaPipe FaceMesh 检测关键点。

### 3. 单张图像处理流程

整体流程如下：

```text
原始 BGR 图像
  → 读取或检测 FaceMesh landmarks
  → 基于固定 landmark index 定义眼、唇、眉、脸轮廓区域
  → 计算各局部 ROI 的 bbox
  → bbox 边界裁剪与最小尺寸检查
  → 裁剪原图对应区域
  → resize 到固定 ROI 尺寸
  → 保存四类 ROI PNG
  → 写入 roi_metadata.csv 和 failed_cases.csv
```

该脚本没有做人脸整体对齐，也没有将双眼映射到统一模板位置；它是在原始图像坐标系下直接基于 FaceMesh landmark 裁剪局部区域。因此，输入图像如果本身存在较大旋转、侧脸、尺度差异或裁剪差异，局部 ROI 也会继承这些几何差异。

### 4. FaceMesh 检测与 landmark 处理

脚本使用 MediaPipe FaceMesh 作为默认关键点检测器：

```python
static_image_mode=True
max_num_faces=1
refine_landmarks=True
min_detection_confidence=0.5
```

如果输入图像最长边超过 `max_detection_side`，脚本会仅将传入 MediaPipe 的检测图缩小到指定尺寸，以降低检测开销；输出 landmark 仍按原图宽高换算回原始图像像素坐标。

landmark 检测后会进行基础质量检查：

- landmark 数量必须至少为 468；
- landmark 坐标必须全部为有限值；
- 如果读取外部 landmark 文件，既支持归一化坐标 `[0,1]`，也支持像素坐标。

如果检测不到人脸、关键点不足或关键点异常，该样本不会生成 ROI，而是写入 `failed_cases.csv`。

### 5. 四类 ROI 的几何定义

#### 5.1 眼周 ROI

眼周 ROI 使用左右眼 FaceMesh landmark 合并计算，不分别裁剪左眼和右眼，也不做左右眼拼接。

主要设计思路：

- 先根据 `LEFT_EYE_INDICES` 和 `RIGHT_EYE_INDICES` 得到双眼区域；
- 再结合 face oval 的宽度确定最终 ROI 宽度；
- 眼周 ROI 的宽度至少达到脸宽的一定比例，避免眯眼、闭眼或 landmark 抖动导致 ROI 过窄；
- 垂直方向不是按眼裂高度裁剪，而是按目标宽高比计算固定高度；
- 下方保留比例大于上方，用于包含下眼睑、眼袋和眶下软组织。

输出尺寸为：

```text
224 × 112
```

该 ROI 的医学意图是保留眼周水肿、眼袋、眼睑状态和眶周皮肤等局部信息。

#### 5.2 唇周 ROI

唇周 ROI 基于嘴唇 landmark 生成 bbox，然后进行横向和纵向扩展：

- 横向扩展比例：`lip_expand_x = 0.35`
- 纵向扩展比例：`lip_expand_y = 0.60`

该区域不只包含红唇边界，而是保留嘴角、口周皮肤、鼻下区域和下唇下方皮肤。

输出尺寸为：

```text
224 × 112
```

该 ROI 的意图是保留唇色、口周形态、口唇闭合状态和相关局部纹理信息。

#### 5.3 双侧脸颊 ROI

脸颊 ROI 使用 face oval、眼部 bbox 和唇部 bbox 共同确定。它不是裁剪一整块面中部区域，而是分别裁剪左脸颊和右脸颊，然后横向拼接。

横向范围按 face width 比例定义：

- 左脸颊：脸宽的 `0.10–0.36`
- 右脸颊：脸宽的 `0.64–0.90`

纵向范围大致从眼部下方开始，到唇部上方或脸高比例限制处结束。脚本还设置了最小脸颊宽度和高度，避免嘴部较宽、头部轻微旋转或 landmark 抖动时 ROI 过小。

左右脸颊各自 resize 到半宽后拼接，最终输出尺寸为：

```text
224 × 224
```

该 ROI 的意图是关注中下脸颊、颧部下方、鼻翼外侧到口角外侧附近的皮肤颜色、纹理和软组织状态。

#### 5.4 额头 ROI

额头 ROI 使用眉毛 landmark 和 face oval 共同确定。它取眉毛上方一条区域，而不是完整额头到发际线区域。

主要设计思路：

- 横向范围基于双眉 bbox，并向左右按脸宽比例扩展；
- 下边界位于眉毛上方，留出小间隔；
- 上边界按脸高比例向上推；
- 设置最小高度，避免 ROI 过薄。

输出尺寸为：

```text
224 × 112
```

额头 ROI 更偏向控制区域，用于保留眉上额头皮肤，同时尽量减少头发、发际线和背景干扰。但如果样本存在刘海遮挡，脚本本身不会自动剔除，需要依赖可视化结果人工检查。

### 6. 输出结构

每个成功样本会生成一个以 patient_id 命名的子目录：

```text
roi_dataset/
  patient_id/
    eye_roi.png
    lip_roi.png
    cheek_roi.png
    forehead_roi.png
```

同时输出两个表格：

```text
roi_metadata.csv
failed_cases.csv
```

`roi_metadata.csv` 中记录：

- `patient_id`
- 原始 `image_path`
- `eye_roi_path`
- `lip_roi_path`
- `cheek_roi_path`
- `forehead_roi_path`
- `nyha_binary`
- `sex`
- 标签表中需要保留的字段，例如 `fold`、`NYHA`、`nyha_grade`、`y`、`label`

`failed_cases.csv` 中记录：

- `patient_id`
- `image_path`
- `reason`

如果运行时指定 `--save_vis`，脚本还会在：

```text
roi_dataset/vis/
```

下保存带有 ROI 框的可视化图，便于检查 landmark 和 bbox 是否合理。

### 7. 该方案保留与抑制的信息

**主要保留的信息：**

- 眼周局部形态和皮肤纹理；
- 嘴唇与口周形态、颜色和纹理；
- 双侧脸颊皮肤颜色、纹理和软组织状态；
- 眉上额头皮肤；
- 一定程度的局部非对称性。

**主要抑制或丢失的信息：**

- 整体脸型和完整头面部轮廓；
- 全局五官空间布局；
- 发型、耳部、颈部、肩部和衣物；
- 大部分原始背景；
- 全局人脸尺度、姿态和上下文信息。

因此，该数据版本更适合回答“局部面部区域是否包含可用于 NYHA 分类的信息”，而不适合作为完整 global face 输入的直接替代。

### 8. 实现上的注意事项与局限

1. **当前脚本不是全局人脸 ROI 提取程序**  
   它不会输出完整人脸 ROI，只输出眼、唇、脸颊、额头四类局部 ROI。如果实验目标是全局人脸输入，需要新增 global face ROI 提取逻辑。

2. **默认项目根目录可能不符合当前仓库结构**  
   脚本中 `PROJECT_ROOT = Path(__file__).resolve().parents[2]`。对于当前路径 `E:\projects\face2\preprocessing\preprocess_single_roi_dataset.py`，`parents[2]` 会指向 `E:\projects`，而不是 `E:\projects\face2`。如果使用默认路径，可能读写到错误目录。更稳妥的做法是运行时显式传入 `--input_image_dir`、`--label_csv_path` 和 `--output_roi_dir`，或将项目根目录修正为 `parents[1]`。

3. **没有做全局对齐或相似变换归一化**  
   脚本只在原图坐标系下直接裁剪局部区域，没有进行眼睛水平校正、统一眼距、统一脸部位置或模板对齐。因此，如果原图姿态差异较大，ROI 的几何一致性会受到影响。

4. **没有使用 face parsing 或语义分割**  
   ROI 是几何 bbox 裁剪，不是语义区域裁剪。头发、遮挡物、阴影、背景或非皮肤区域可能进入 ROI，尤其是额头和脸颊区域。

5. **只处理检测到的第一张人脸**  
   MediaPipe 设置 `max_num_faces=1`。如果原图包含多个人脸，脚本默认使用检测到的第一张脸，没有按面积、中心位置或 ID 进一步筛选。

6. **ROI 失败会按样本记录，但不会中断整体流程**  
   单个样本失败后写入 `failed_cases.csv`，其余样本继续处理。后续训练应以 `roi_metadata.csv` 中成功样本为准。

7. **重复运行不会自动清空旧输出**  
   如果输出目录已存在，脚本只会在指定 `--overwrite` 时覆盖同名 ROI 文件，但不会主动清理已废弃样本目录。严谨实验中应为每次数据版本使用独立输出目录，或运行前手动清理。

8. **输出为 PNG，但未执行模型输入归一化**  
   脚本只保存 `uint8` BGR 图像文件，不执行 ImageNet Normalize、VGGFace2 mean subtraction 或其他模型输入归一化。归一化应在 Dataset 或训练 transform 阶段完成。

### 9. 与全局人脸 ROI 方案的关系

如果下一步目标是构建全局人脸 ROI，该脚本可以复用以下基础组件：

- `load_image()`：稳健读取图像；
- `detect_face_landmarks()`：检测或读取 FaceMesh landmarks；
- `clamp_bbox()`、`expand_bbox()`、`ensure_min_bbox_size()`：bbox 边界和尺寸控制；
- `crop_and_resize()`：裁剪并 resize；
- `save_png()`：Windows 路径友好的 PNG 保存；
- `save_visualization()`：ROI 可视化质控；
- `roi_metadata.csv` / `failed_cases.csv` 的日志设计。

但全局人脸 ROI 的核心逻辑应重新定义，例如：

```text
原图
  → FaceMesh 检测
  → 基于 FACE_OVAL_INDICES 得到全脸 bbox
  → 按比例扩展额头、脸侧和下巴
  → 可选：基于双眼做相似变换对齐
  → 裁剪整张人脸 ROI
  → resize 到 224×224
  → 保存 global_face_roi.png
```

如果希望与当前 `global_face_parsing_hybrid_foreheadrepair_blackbg_224_png_strict` 数据保持一致，则更建议复用已有全局人脸预处理流程，而不是直接改造该局部 ROI 脚本。

### 10. 总结

`preprocess_single_roi_dataset.py` 构建的是一种局部 ROI 数据版本。它通过 FaceMesh landmarks 在原图坐标系中裁剪眼周、唇周、双侧脸颊和额头四类区域，并将每类 ROI resize 到固定尺寸保存为 PNG。该方案的价值在于隔离局部解剖区域，评估模型是否能从局部面部特征中学习 NYHA 相关信号。

该方案的核心特点可以概括为：

```text
FaceMesh landmark 定位
  + 局部解剖区域 bbox 规则
  + 固定尺寸 ROI 输出
  + metadata / failed cases / 可视化质控
  = 局部面部区域数据集
```

它与全局人脸数据集互补：全局人脸保留整体头面部结构和空间布局，局部 ROI 则更强调特定区域的颜色、纹理和形态信息。后续如果要提取“全局人脸 ROI”，应在该脚本基础上新增独立的 global face ROI 逻辑，而不是直接使用现有四类局部 ROI 输出。
## `build_global_face_parsing_hybrid_foreheadrepair_blackbg_224_png_strict.py`：原图坐标直接对齐 + 语义脸部掩膜 + 额头修复黑背景 PNG

### 一、这个预处理程序要解决的问题

该程序用于生成 224×224 的全局人脸图像，输出形式是普通 RGB PNG，背景统一置为黑色。它的核心目标不是简单裁剪人脸，而是尽量保留真正属于“脸部”的区域，同时去除背景、头发、耳朵、颈部、衣服和肩部等非面部信息，减少模型从无关背景或拍摄环境中学习偏差。

它相对于较早的规则椭圆或规则 envelope 掩膜方案，有两个重点改进：

1. 对齐阶段不再把检测框裁剪图作为最终变换来源，而是在检测框内运行 FaceMesh 后，把关键点坐标还原到原始图像坐标系，再直接从原图进行仿射变换。这样可以减少旋转对齐后由裁剪框边界带来的黑色切边。
2. 最终掩膜不默认使用规则椭圆或大范围规则 envelope，而是以 BiSeNet 人脸解析得到的语义脸部区域为主体；只有在额头区域疑似被头发遮挡、语义分割切掉额头时，才在局部额头带内引入规则 envelope 进行修复。

因此，这个程序生成的是“直接从原图对齐后的语义脸部黑背景图”，更强调保留自然脸型和语义面部结构，而不是生成规则化椭圆脸。

### 二、输入、配置与输出位置

程序默认读取配置文件：

```text
config/preprocess/global_face_parsing_hybrid_foreheadrepair_blackbg_224_png_strict.yaml
```

当前配置中的关键路径是：

```text
split_csv: data\raw\label_raw.csv
image_dir: data/raw/images
output_dir: data/processed/global_face/global_face_parsing_hybrid_foreheadrepair_blackbg_224_png_strict
```

主要参数包括：

- `image_size: 224`：最终输出图像大小为 224×224。
- `min_detection_confidence: 0.5`：MediaPipe 人脸检测和关键点检测的最低置信度。
- `parsing_model: bisenet`：使用 BiSeNet 做人脸语义解析。
- `parsing_checkpoint: preprocessing/checkpoints/face_parsing/79999_iter.pth`：BiSeNet 权重路径。
- `final_mask_mode: hybrid`：最终掩膜采用语义掩膜为主、额头局部修复为辅的混合策略。
- `forehead_band_ratio: 0.35`：额头修复只允许发生在图像顶部约 35% 的人脸区域内。
- `hair_repair_threshold: 0.10`：候选修复区域内头发比例超过该阈值时，触发额头修复。
- `enable_jaggedness_trigger: false`：当前配置关闭了额头轮廓锯齿触发，仅主要依赖头发遮挡比例触发。
- `feather_kernel: 11`：最终掩膜边缘羽化核大小。
- `overwrite: false`：默认不覆盖已有输出目录。

输出目录下会生成：

- `images/`：最终 224×224 黑背景 RGB PNG 图像。
- `logs/preprocess_log.csv`：逐样本处理日志。
- `logs/failed_cases.csv`：失败样本列表。
- `logs/parsing_hybrid_summary.csv`：按成功、失败、掩膜质量、修复类型等统计的汇总表。
- `logs/preprocess_summary.txt`：文本版处理摘要。
- `qc_preview/`：质控预览图，包含随机成功样本、语义掩膜样本、额头修复样本、高头发/高衣服/高颈部占比样本、低/高掩膜面积样本和失败样本等分类。

### 三、整体处理流程

整体流程可以概括为：

```text
读取 label_raw.csv 中的 ID
→ 在 raw/images 中按 ID 查找原始图片
→ MediaPipe FaceDetection 检测人脸框
→ 选择主脸
→ 在扩展后的人脸框内运行 FaceMesh
→ 将 FaceMesh 关键点从裁剪框坐标还原到原图坐标
→ 基于原图坐标估计相似变换
→ 直接从原始 RGB 图像 warp 到 224×224 标准模板
→ 对对齐后图像运行 BiSeNet 人脸解析
→ 提取皮肤、眉毛、眼睛、鼻子、嘴唇/嘴部等语义脸部类别
→ 对语义脸部 mask 做轻量形态学规则化
→ 构造候选规则 envelope，仅用于额头修复判断
→ 根据头发遮挡比例决定是否在额头局部合并 envelope
→ 对最终 mask 羽化
→ 使用黑色背景合成最终 RGB 图像
→ 保存 PNG、日志和 QC 预览
```

这个流程的关键点是：人脸对齐来自原图，掩膜主体来自语义解析，规则 envelope 只作为局部额头修复工具，而不是最终脸型模板。

### 四、原图坐标直接对齐

程序先用 MediaPipe FaceDetection 在原图中检测人脸框，然后选取主脸。随后程序会扩展检测框，在扩展后的人脸 crop 上运行 MediaPipe FaceMesh。FaceMesh 的检测发生在 crop 内，但程序会把关键点坐标重新映射回原始图像坐标系。

之后，程序根据这些原图坐标系下的关键点，估计到 224×224 标准人脸模板的相似变换矩阵，并直接对原始 RGB 图像执行 `cv2.warpAffine`。

这种设计的意义是：

- FaceMesh 在局部人脸 crop 上运行，关键点检测更稳定。
- 最终图像却直接从原图变换得到，不以 crop 图像作为最终来源。
- 旋转或缩放时，如果目标区域超过原始图像边界，才会出现黑色填充；不会因为中间裁剪框过小而产生额外的旋转切边。
- 程序会记录 `top_invalid_area_ratio` 和 `top_cut_warning`，用于发现顶部无效黑边较多的样本。

因此，这一步主要解决的是“对齐质量”和“裁剪边界黑边伪影”的问题。

### 五、BiSeNet 人脸解析与语义脸部掩膜

对齐后的 RGB 图像会送入 BiSeNet 人脸解析模型。推理前程序会把图像 resize 到解析模型输入大小，并使用 ImageNet 均值和标准差做归一化。需要注意的是，这个归一化只用于 BiSeNet 推理，最终保存的 PNG 仍然是普通 `uint8` RGB 图像，不是归一化后的张量。

程序选择的语义脸部类别主要包括：

- 皮肤
- 左右眉毛
- 左右眼睛
- 鼻子
- 嘴部
- 上唇和下唇

程序不会把头发、耳朵、颈部、衣服、背景等类别纳入基础语义脸部掩膜。这样可以减少模型利用发型、衣领、肩部、拍摄背景等非脸部因素进行分类的风险。

得到初始语义 mask 后，程序还会做轻量规则化：

- 二值化语义脸部区域。
- 面积过小时判为失败。
- 使用小核形态学闭运算连接局部断裂区域。
- 只保留最大连通域。
- 只填补较小的内部空洞，避免把大块非脸部区域误填进去。
- 使用轻度高斯平滑后再阈值化，使边缘更连续。

这一步的原则是“修正明显噪声，但不把脸强行改成规则椭圆”。因此输出 mask 仍然尽量保留自然脸型和语义边界。

### 六、Hybrid 额头修复策略

该程序的最终掩膜模式为 `hybrid`。在这个模式下，最终 mask 默认等于规则化后的语义脸部 mask。程序同时会生成一个候选规则 envelope，但它不是默认最终 mask，而是只用于额头局部修复。

候选 envelope 的扩展参数包括：

- `forehead_expand_ratio: 0.18`
- `side_expand_ratio: 0.05`
- `chin_expand_ratio: 0.03`

程序会统计候选 envelope 内的头发比例。如果头发遮挡比例超过 `hair_repair_threshold: 0.10`，则认为额头区域可能被头发语义干扰，导致原始语义脸部 mask 把部分额头切掉。此时程序只在顶部额头带内，把候选 envelope 与语义 mask 做局部合并。

当前配置中 `enable_jaggedness_trigger: false`，所以额头轮廓锯齿度虽然会被计算，但不会作为触发修复的主要条件。实际触发主要来自头发遮挡比例。

这种 hybrid 设计的优点是：

- 大多数样本保持语义解析得到的自然脸部边界。
- 只有疑似刘海、头发遮挡导致额头缺失时，才局部补偿。
- 不会像全局椭圆或全局 envelope 那样，把头发、耳朵、衣领、颈部大面积纳入最终图像。

其本质是“语义 mask 优先，规则 mask 只做额头局部纠偏”。

### 七、黑背景合成与 PNG 保存

最终 mask 会通过 `feather_kernel: 11` 做羽化，得到 0 到 1 之间的 alpha。随后程序将对齐后的 RGB 图像与黑色背景合成：

```text
final_rgb = aligned_rgb * alpha + black * (1 - alpha)
```

因为背景是纯黑色，所以最终图像只保留羽化后的脸部区域，其他区域为黑色。输出图像保存为普通 RGB PNG，可直接作为后续训练输入。训练阶段如果需要 ImageNet normalization，应由训练 transform 再做，而不是依赖该预处理脚本提前保存归一化图像。

### 八、质量控制与日志信息

程序对每个样本记录较多质量控制信息，包括：

- 是否成功处理。
- 失败原因。
- 原始图像尺寸。
- 检测框位置和大小。
- 对齐后的无效区域比例。
- 最终 mask 面积比例。
- 语义 mask 面积比例。
- final mask 内头发、颈部、衣服、背景等非目标类别比例。
- 是否触发额头修复。
- 修复原因，例如 `hair_occlusion`。
- 是否出现高头发比例、高衣服比例、高颈部比例、低 mask 面积、高 mask 面积、顶部切边等 warning。

QC 预览图通常包含多个面板，用于人工检查：

```text
原图
检测框/裁剪区域
直接对齐图
parsing label map
selected semantic mask
regularized semantic mask
candidate envelope
final hybrid mask
最终黑背景 PNG
```

因此，这个程序不只是生成训练图像，也提供了比较完整的失败样本和可疑样本追踪机制。

### 九、失败策略

该程序属于较严格的预处理流程。以下情况通常会被记录为失败，而不是强行生成结果：

- 找不到对应图片。
- 图片读取失败。
- 未检测到人脸。
- FaceMesh 关键点不足或无法估计稳定变换。
- 仿射对齐失败。
- BiSeNet 人脸解析失败。
- 语义 mask 或最终 mask 面积过小。
- PNG 保存失败。
- 其他未捕获异常。

这种严格策略的好处是输出数据质量更可控；代价是会丢弃部分难处理样本，需要后续查看 `failed_cases.csv` 和 QC 预览，判断是否需要手工修正、降低阈值或单独处理。

### 十、该预处理方案保留和抑制的信息

该方案主要保留：

- 对齐后的整体人脸结构。
- 皮肤区域。
- 眉毛、眼睛、鼻子、嘴部和嘴唇等面部语义结构。
- 少量经额头修复补回的额头区域。

该方案主要抑制：

- 原始拍摄背景。
- 头发和发型信息。
- 耳朵。
- 颈部。
- 衣服、衣领和肩部。
- 中间裁剪导致的旋转黑边伪影。

从建模角度看，它更适合希望模型关注面部形态、颜色和局部语义结构，而不是依赖背景、衣服、拍摄环境或发型的实验。

### 十一、需要注意的局限性

1. 该流程依赖 BiSeNet checkpoint。如果 `preprocessing/checkpoints/face_parsing/79999_iter.pth` 缺失，无法完成人脸解析。
2. BiSeNet 的语义解析错误会直接影响最终 mask。例如把头发误判为皮肤，或把额头皮肤误判为头发，都会影响最终输出。
3. 额头修复是启发式策略，主要由候选 envelope 内头发比例触发。它可以修复部分刘海遮挡导致的额头缺失，但不能保证所有样本都正确。
4. 当前配置关闭了 jaggedness trigger，因此额头轮廓锯齿度只作为记录信息，不作为主要修复触发条件。
5. 黑背景本身是一种人为构造的统一背景。虽然它减少了自然背景差异，但也会引入“黑色背景”这一固定分布。
6. 输出 PNG 没有做训练归一化。模型训练时仍需要在 dataset transform 中完成 resize、tensor 转换和 normalization。
7. `overwrite: false` 时，如果输出目录已存在，程序会拒绝继续写入；需要明确传入覆盖参数或清理输出目录后再运行。
8. 顶部切边只作为 warning 记录，不一定导致样本失败，因此仍需要结合 QC 图人工检查严重样本。

### 十二、与其他预处理结果的关系

该程序生成的目录：

```text
data/processed/global_face/global_face_parsing_hybrid_foreheadrepair_blackbg_224_png_strict
```

就是当前项目中“global face + parsing hybrid + forehead repair + black background + 224 PNG + strict”这一类全局脸部预处理数据。它不是 ROI 局部区域数据，也不是椭圆 mask 或均值背景版本。

如果后续实验使用该目录作为输入，应在实验命名或配置中明确标注其特点：

```text
Global224 / parsing hybrid / forehead repair / black background / strict
```

这样可以避免把它与全局椭圆脸、ROI 单区域、多 ROI 拼接、均值背景或普通裁剪版本混淆。
